Context

1 Campaign Roles in Price Reject Setups

One application generates multiple attempts, one per campaign pinged. Some campaigns participate in price reject setups where a top-tier campaign places a bid, and a negotiable "catcher" campaign fulfills it.

The tier_campaign_buffer table links negotiable campaigns to their top-tier parent via parent_campaign_id.

3 The Conversion

The conversion records campaign_id = C_150 (the negotiable), not C_100 (the top-tier that bid). This mismatch is the root cause of the join problem.

The Problem

4 The Current Join is Broken

The existing query joins on parent_campaign_id, which is C_100 for both the top-tier and the negotiable. The conversion's campaign_id is C_150. The condition parent_campaign_id = conversion.campaign_id never matches for price reject setups, producing zero revenue for all attempts.

LEFT JOIN conversion_cte   ON attempt.application_id = conversion.application_id   AND attempt.parent_campaign_id = conversion.campaign_id

The Decision
Which join strategy should replace the broken one?
Two viable alternatives. Select an approach and dismiss the other with a note.
Option B: Join on (application_id, campaign_id)
Recommended PRECISE
Adding attempt.campaign_id = conversion.campaign_id restricts the join to only the attempt whose campaign completed the conversion. Only the negotiable (C_150) matches, and its parent_campaign_id (C_100) correctly receives the revenue in Stage 2 aggregation.
LEFT JOIN conversion_cte   ON attempt.application_id = conversion.application_id   AND attempt.campaign_id = conversion.campaign_id

OVERCOUNTS PRODUCTION ETL
Joining only on application_id matches every attempt for the same application. Revenue spreads to campaigns that had nothing to do with the sale. The production ETL uses this approach because the overcount was acceptable for its reporting use case.
